# Azure Cloud Hierarchy

Before understanding logs, we must understand **how Azure organizes
environments**.

Azure uses a structured hierarchy.

    Tenant
    └ Subscription
       └ Resource Group
          └ Resource

This hierarchy defines:

-   identity boundaries
-   access control
-   billing
-   infrastructure organization

------------------------------------------------------------------------

# 1. Azure Tenant

### Concept

A **Tenant** is the **top-level identity container in Azure**.

It represents the **entire organization's cloud identity environment**.

Example:

    companyname.onmicrosoft.com

The tenant is managed by:

**Microsoft Entra ID (Azure AD)**

### What Exists Inside a Tenant

A tenant contains:

-   Users
-   Groups
-   Applications
-   Policies
-   Azure Subscriptions

Example objects:

    Users
    Groups
    Enterprise Applications
    Conditional Access Policies

### Security Importance

Tenant is critical because **identity is the primary attack surface in
cloud environments.**

Many attacks target:

-   user accounts
-   admin roles
-   authentication tokens

Logs from tenant authentication activity are stored in:

    SigninLogs
    AuditLogs

These logs are heavily used in **SOC investigations.**

------------------------------------------------------------------------

# 2. Azure Subscription

### Concept

A **subscription is a billing and resource boundary.**

Organizations often separate subscriptions by environment.

Example:

    Production
    Security
    Development
    Testing

### Why Subscriptions Exist

They allow organizations to:

-   isolate workloads
-   manage budgets
-   apply security policies
-   control resource access

Example structure:

    Tenant
     ├ Production Subscription
     ├ Security Subscription
     └ Development Subscription

------------------------------------------------------------------------

# 3. Resource Group

### Concept

A **Resource Group is a logical container for Azure resources.**

It groups related infrastructure together.

Example:

    RG-SOC
    RG-Logging
    RG-Production-VMs

### Why Resource Groups Exist

They help manage:

-   deployment
-   permissions
-   lifecycle

Example:

    RG-SOC
     ├ Log Analytics Workspace
     ├ Microsoft Sentinel
     ├ Automation Account

------------------------------------------------------------------------

# 4. Resource

Resources are the **actual Azure services running in the cloud.**

Examples:

    Virtual Machine
    Storage Account
    Key Vault
    Log Analytics Workspace
    Microsoft Sentinel

From a SOC perspective, the **most important resource is Log Analytics
Workspace**.

------------------------------------------------------------------------

# Log Analytics Workspace

## Concept

Log Analytics Workspace is the **central data storage for security
telemetry in Azure Monitor.**

It stores logs generated by:

-   Microsoft Defender
-   Entra ID
-   Azure Activity Logs
-   Microsoft 365
-   Windows Security Logs
-   Linux Syslog
-   Network devices

It acts as the **database used by Microsoft Sentinel SIEM.**

------------------------------------------------------------------------

# Why Log Analytics Exists in Enterprise Security

Enterprise SOC teams must monitor activity across many systems:

    Endpoints
    Identity systems
    Email systems
    Cloud infrastructure
    Applications

Each system generates logs.

Without a **central log platform**, investigation would be impossible.

Log Analytics solves this problem by providing:

-   centralized log collection
-   large-scale log storage
-   powerful query engine (KQL)
-   integration with SIEM detections

# Log Storage Model

Logs are stored in **structured tables**.

Each table contains:

    TimeGenerated
    EventData
    Entities
    Metadata

Example record:

    TimeGenerated: 2026-03-05T08:15
    User: john.doe
    IPAddress: 45.22.11.9
    Action: LoginFailure

------------------------------------------------------------------------

# KQL Query Engine

SOC analysts search logs using **Kusto Query Language (KQL)**.

Example query:

    SigninLogs
    | where ResultType != 0
    | project TimeGenerated, UserPrincipalName, IPAddress

KQL is used for:

-   threat hunting
-   detection engineering
-   investigations
-   SIEM rules

# Important Log Tables

SOC analysts investigate security incidents using specific tables.

------------------------------------------------------------------------

# SigninLogs

**Source:** Microsoft Entra ID

Tracks authentication activity:

    Successful logins
    Failed logins
    MFA attempts
    Conditional access evaluations

Key Fields:

    UserPrincipalName
    IPAddress
    Location
    ResultType
    AppDisplayName

Detection Use Cases:

-   brute force attacks
-   credential stuffing
-   impossible travel
-   suspicious login locations

------------------------------------------------------------------------

# SecurityEvent

**Source:** Windows Security Logs

Collected from:

    Domain Controllers
    Windows Servers
    Endpoints

Example Events:

    4624 – Successful login
    4625 – Failed login
    4720 – User account created
    4672 – Admin privileges assigned

Used to detect:

-   account compromise
-   privilege escalation
-   lateral movement
-   suspicious logins

------------------------------------------------------------------------

# AzureActivity

**Source:** Azure Control Plane Logs

Tracks:

    Virtual machine creation
    Resource deletion
    Role assignments
    Network changes

Security Use Cases:

    Privilege escalation
    Unauthorized admin activity
    Infrastructure tampering

------------------------------------------------------------------------

# DeviceEvents

**Source:** Microsoft Defender for Endpoint

Tracks:

    File activity
    Registry changes
    System behavior

Used to detect malware and suspicious behavior.

------------------------------------------------------------------------

# DeviceProcessEvents

Tracks process execution on endpoints.

Examples:

    powershell.exe
    cmd.exe
    rundll32.exe
    wmic.exe

Important Fields:

    ProcessName
    ProcessCommandLine
    InitiatingProcessName
    DeviceName
    AccountName

Detection Use Cases:

-   PowerShell attacks
-   encoded commands
-   malware execution

------------------------------------------------------------------------

# OfficeActivity

**Source:** Microsoft 365 Audit Logs

Tracks activity in:

    Exchange
    SharePoint
    OneDrive
    Teams

Detects:

    Mailbox rule creation
    Mass downloads
    Data exfiltration

# Example Detection

### Brute Force Login Detection

    SigninLogs
    | where ResultType != 0
    | summarize FailedAttempts=count() by IPAddress, bin(TimeGenerated,5m)
    | where FailedAttempts > 10

Detection logic:

    Failed login attempts
    ↓
    Aggregate per IP
    ↓
    Threshold exceeded
    ↓
    Sentinel alert
