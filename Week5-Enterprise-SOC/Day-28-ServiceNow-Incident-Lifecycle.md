# Day 28 – ServiceNow Incident Lifecycle (Enterprise SOC)

## Objective

Understand how alerts generated in Microsoft Sentinel are converted into **trackable, accountable incidents inside ServiceNow**, and how SOC teams manage incidents using priority, severity, SLA, and assignment.

This is the **final operational layer** of the SOC pipeline where security events become **business-impacting tickets**. 

---

# 1. Concept Overview

ServiceNow is an **IT Service Management (ITSM) platform** used by SOC teams to:

* Track security incidents
* Assign ownership
* Enforce SLAs
* Maintain audit trails
* Manage escalation workflows

### Core Flow

```
Alert (Sentinel)
↓
Incident (Sentinel)
↓
ServiceNow Ticket
↓
SOC Investigation
↓
Resolution + Closure
```

This is where **security meets operations**.

---

# 2. Why This Exists in Enterprise Security

SIEM tools (like Sentinel) detect threats — but enterprises need:

* Accountability (who owns the incident?)
* Tracking (what is the status?)
* SLA enforcement (how fast must we respond?)
* Reporting (metrics for management)

Without ServiceNow:

* Alerts remain unmanaged
* No ownership → incidents get ignored
* No SLA → delayed response → higher breach impact

👉 ServiceNow converts **security signals → operational responsibility**

# 4. Core Components

## 4.1 Alert

* Generated by Sentinel detection rule
* Represents a **single suspicious activity**

## 4.2 Incident

* Group of related alerts
* Provides context:

  * User
  * IP
  * Device
  * Timeline

## 4.3 ServiceNow Ticket

* Created via integration (API / Logic Apps / Playbooks)
* Tracks:

  * status
  * ownership
  * SLA
  * actions taken

---

# 5. Key Fields (CRITICAL FOR SOC)

## 5.1 Priority

Defines **how urgent the ticket is**.

```
Priority = Impact + Urgency
```

Example:

| Impact | Urgency | Priority |
| ------ | ------- | -------- |
| High   | High    | P1       |
| High   | Medium  | P2       |
| Low    | Low     | P4       |

👉 Used for **queue sorting and escalation**

---

## 5.2 Severity

Defined by detection rule in Sentinel:

* Critical
* High
* Medium
* Low

### Difference from Priority:

| Severity       | Priority         |
| -------------- | ---------------- |
| Technical risk | Business urgency |

Example:

* Malware on Domain Controller → High Severity + P1
* Failed login alert → Medium Severity + P3

---

## 5.3 SLA (Service Level Agreement)

Defines **time-bound response requirements**.

Example:

| Severity | SLA      |
| -------- | -------- |
| Critical | 15 min   |
| High     | 30 min   |
| Medium   | 2 hours  |
| Low      | 24 hours |

If SLA is violated:

* Escalation triggered
* Management notified

---

## 5.4 Assignment

Defines **who owns the ticket**

Types:

* Tier 1 SOC (L1)
* Tier 2 SOC (L2)
* Incident Response Team
* Cloud Security Team

# 8. Detection Logic Context

ServiceNow does NOT detect threats.

Instead it handles:

* **Post-detection workflow**
* **Operational response**

However, detection logic affects ticketing:

### Example

KQL Detection:

```
SigninLogs
| where ResultType != 0
| summarize FailedAttempts=count() by IPAddress, bin(TimeGenerated,5m)
| where FailedAttempts > 10
```

This generates:

* Alert → Incident → ServiceNow Ticket

# 9. Investigation Workflow (SOC Perspective)

## Step 1 – Ticket Intake

L1 Analyst checks:

* Priority
* Severity
* Affected user/device
* Alert summary

---

## Step 2 – Validate Alert

Questions:

* Is this real suspicious activity?
* Known IP or attacker IP?
* User behavior normal?

---

## Step 3 – Deep Investigation

Use:

* SigninLogs
* DeviceProcessEvents
* AzureActivity

Check:

* Timeline of events
* Correlated alerts
* Lateral movement

---

## Step 4 – Decision

* False Positive → Close ticket
* True Positive → Escalate to L2

---

## Step 5 – Response

* Disable account
* Block IP
* Isolate device

---

## Step 6 – Documentation

Update ticket with:

* Findings
* Evidence
* Actions taken

* Security scans

# 13. Tuning Strategy

Improve ticket quality by:

* Adjusting detection thresholds
* Excluding trusted IPs
* Filtering service accounts
* Reducing noise alerts

Goal:

👉 Fewer tickets → Higher quality alerts

# 15. Interview Talking Points

1. ServiceNow is used to convert security incidents into **trackable operational tickets**.
2. Alerts → Incidents → Tickets ensures **accountability and SLA enforcement**.
3. Priority is based on **business impact + urgency**, while severity is technical.
4. SOC analysts work tickets based on **priority queues and SLA deadlines**.
5. ServiceNow integrates with Sentinel via **playbooks/API for automated ticket creation**.
